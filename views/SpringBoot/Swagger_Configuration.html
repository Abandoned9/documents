    # 使用Swagger来管理基于SpringBoot项目的API

    在常用的前后端分离项目中，后端一般只提供一个RESTful类型的API接口

    # 前后端分离模式简要阐述

    目前的前后端分离模式：

    • 后端：后端控制层(Contorller)，服务层(Service)，数据访问层(API接口)

    • 前端：前端控制层，视图层

    后端通过一个地址发送一串JSON的数据，前端通过这个地址获取一串JSON数据然后在视图层展示数据，这就是前后端的交互模式。这种设计概念相对独立并且是松耦合，前端和后端甚至不需要部署在同一台服务器上，这样也能缓解单个服务器的压力。

    # 前后端分离模式产生的问题

    前后端分离之后，前端和后端两个团队开发，如果使用Word或者其他文档软件来逐个记录后端所开放的API接口，这样不仅降低了团队工作效率，沟通效率，还有可能积攒种种问题。

    在项目开发中，首先会指定一个计划提纲(schema)，事实更新最新的API，降低集成项目时会造成的风险，但是当API接口数量变得繁杂，查阅和使用就成了问题。

    那么如何使用Swagger来方便快捷的管理后端发送过来的接口呢

    # Swagger介绍

    Swagger是一个RESTful API接口自动生成工具，它能让API文档与API定义同步更新，当后端修改API之后，只需要启动后端项目，API接口就会自动扫描并且同步。他支持Java，PHP等目前流行的开发语言...

    Swagger官方网站 (opens new window)

    目前最新版本的Swagger已经在使用OAS3.0协议规范，这里是具体协议 (opens new window)

    # Swagger通过Maven集成到SpringBoot

    在Maven项目的pom.xml里的<dependencies>标签内添加

    <dependency>
    <groupId>io.springfox</groupId>
    <artifactId>springfox-boot-starter</artifactId>
    <version>3.0.0</version>
</dependency>

    然后在SpringBootApplication类统计目录下创建一个config包，通常用来放各个框架的配置文件，然后新建一个SwaggerConfig类，打上@Configuration注解等等巴拉巴拉...

    ​ 注意，在Swagger2.0版本中，需要在SwaggerConfig类上添加@EnableSwagger2来告诉Swagger这是它的配置类，但是在3.0版本中，只需要在SpringBoot启动类上使用@EnableOpenApi来开启OAS即可

    @Controller

    # 配置Swagger扫描指定的接口

    如上图所示，除了我定义的两个controller之外，swagger还自动扫描到了一个SpringBoot自带的error的控制层 (这个也是可以自定义的，以后可能会专门提出来说说)。

    # apis()

    这里提示了apis需要一个RequestHandle，输入request，它自动提示生成了

    apis(requestHandler -> )
    1

    这是一个Lambda，说明这是一个接口，我们这里直接用它的实体类，也就是配置那些接口能被扫描到的方式，我们改成这样：

    @Bean
public Docket docket(){
    return new Docket(DocumentationType.OAS_30)
            .apiInfo(apiInfo())
            .select()
            .apis(RequestHandlerSelectors.basePackage())
            .build();
}

    在这里我们可以看到有几个关键的后续方法

    • any()
    • basePackage()
    • none()
    • withClassAnnotation()
    • withMethodAnnotation()

    先说说这个basePackge，它是指定我们扫描那个包下面的controller，我们Ctrl点进去这个方法发现它需要的参数是一个字符串，那么我们就指定一下包

    基于我的项目结构，我的例子:

    apis(RequestHandlerSelectors.basePackage("com.dimitri.shirospringboot.controller"))
    1

    指定了之后，Swagger只会扫描controller下面的接口，那么basic-error-controller就不存在了=。=

    any()：扫描所有的controller

    none()：关闭Swagger的日志记录

    withClassAnnotation() 和 withMethodAnnotation() 通过它字面意思就可以理解，一个是通过类上面的Annotation(注解)去扫描，一个是通过方法上面的注解去扫描，那么我们该怎么指定参数？

    点开RequestHandlerSelectors的源码，找到这两个方法所需要的参数：

    public static Predicate<RequestHandler> withMethodAnnotation(final Class<? extends Annotation> annotation) {
    return (input) -> {
        return input.isAnnotatedWith(annotation);
    };
}

public static Predicate<RequestHandler> withClassAnnotation(final Class<? extends Annotation> annotation) {
    return (input) -> {
        return (Boolean)declaringClass(input).map(annotationPresent(annotation)).orElse(false);
    };
}

    # paths()

    paths这个方法相较于apis更加简单，只需要配置你的API接口即可

    点进源码发现它需要一个名为selector，类型为Predicate<String>的参数

    public ApiSelectorBuilder paths(Predicate<String> selector) {
  pathSelector = pathSelector.and(selector);
  return this;
}

    在paths里，最常用的一个方法就是PathSelectors，所以我们直接输入

    发现在PathSelectors有四个方法

    • ant() -> 过滤指定API接口
    • any() -> 过滤所有API接口
    • none() -> 扫描所有API接口
    • regex() -> 过滤所有符合正则表达式的API接口

    例如ant()方法

    .paths(PathSelectors.ant("/user/**"))
    1

    这代表过滤user下所有API接口，即user下的所有API接口都无法被扫描到。

    # Select a definition分组->配置API分组

    在public class Docket implements DocumentationPlugin{}类里面我们可以看到有一个

    public static final String DEFAULT_GROUP_NAME = "default";名为DEFAULT_GROUP_NAME的静态变量，它的默认值为default，也就是我们swaggerUI界面上面的默认组

    我们可以直接在SwaggerConfig的Docket实例中配置，只需要使用groupName()这个方法即可

    例如：

    @Bean
public Docket docket(){
    return new Docket(DocumentationType.OAS_30)
            .apiInfo(apiInfo())
            .groupName("这是一个分组")
            .select()
            .apis(RequestHandlerSelectors.withClassAnnotation(GetMapping.class))
            .paths(PathSelectors.ant("/user/**"))
            .build();
}
